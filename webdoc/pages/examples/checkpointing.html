Checkpoint & Resume

Advanced

Learn to save and restore execution state, implement time-travel debugging, and create branching workflows with checkpoints in rLangGraph.

20 min read

Overview

Checkpointing is one of rLangGraph's most powerful features. It automatically saves the state of your workflow after each step, enabling:

  • Resume from failures - Continue execution after crashes or errors
  • Time-travel debugging - Step backwards through execution history
  • Branching workflows - Create alternative timelines from any checkpoint
  • Human-in-the-loop - Pause for approval and resume later
  • Multi-session support - Save conversations across sessions

Key Concept: rLangGraph creates a checkpoint after every superstep in the Pregel execution model. This provides natural savepoints for persistence.

Why Checkpoints?

Production Reliability

In production, things fail. Network issues, API rate limits, server crashes - checkpoints ensure you never lose progress. Your agent can pick up exactly where it left off.

Development & Debugging

During development, checkpoints let you replay executions, step backwards, and understand exactly what happened at each step.

User Experience

For long-running workflows, users can close the app and return later. Their progress is preserved across sessions.

Checkpoint Savers

rLangGraph provides three built-in checkpoint savers:

MemoryCheckpointSaver

Use for: Testing, development, single-session workflows

Stores checkpoints in memory. Fast but not persistent - lost when process ends. 

use langgraph_checkpoint::MemoryCheckpointSaver;

let checkpointer = MemoryCheckpointSaver::new();

SqliteCheckpointSaver

Use for: Production (single-user), local development, small-scale deployments

Stores checkpoints in a SQLite database. Persistent, file-based storage. 

use langgraph_checkpoint::SqliteCheckpointSaver;

let checkpointer = SqliteCheckpointSaver::new("./checkpoints.db").await?;

PostgresCheckpointSaver

Use for: Production (multi-user), distributed systems, cloud deployments

Stores checkpoints in PostgreSQL. Fully persistent, supports multiple concurrent users. 

use langgraph_checkpoint::PostgresCheckpointSaver;

let checkpointer = PostgresCheckpointSaver::new(
    "postgresql://user:pass@localhost/db"
).await?;

Basic Example

Here's a simple workflow with SQLite checkpointing: 

use langgraph_core::{StateGraph, Message, GraphError};
use langgraph_checkpoint::SqliteCheckpointSaver;
use serde::{Deserialize, Serialize};

#[derive(Clone, Serialize, Deserialize)]
struct State {
    messages: Vec<Message>,
    step: u32,
}

async fn process_step(mut state: State) -> Result<State, GraphError> {
    state.step += 1;
    state.messages.push(Message::assistant(
        format!("Completed step {}", state.step)
    ));

    // Simulate some work
    tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;

    Ok(state)
}

#[tokio::main]
async fn main() -> Result<(), GraphError> {
    // Create SQLite checkpoint saver
    let checkpointer = SqliteCheckpointSaver::new("./checkpoints.db").await?;

    // Build graph
    let mut graph = StateGraph::new();
    graph.add_node("process", process_step);
    graph.add_edge("__start__", "process");
    graph.add_edge("process", "__end__");

    // Compile with checkpointer
    let compiled = graph.compile().with_checkpointer(checkpointer);

    // Create a thread ID for this conversation
    let thread_id = "conversation-123";

    // Initial execution
    let initial_state = State {
        messages: vec![Message::human("Start processing")],
        step: 0,
    };

    let config = CheckpointConfig::new()
        .with_thread_id(thread_id);

    let result = compiled.invoke(initial_state, config).await?;

    println!("Final state: step = {}", result.step);
    // Checkpoint automatically saved!

    Ok(())
}

Resuming Execution

To resume from a checkpoint, use the same thread ID: 

// Resume from last checkpoint
let config = CheckpointConfig::new()
    .with_thread_id("conversation-123");

// This will load the last state and continue
let new_state = State {
    messages: vec![Message::human("Continue from where we left off")],
    step: 0,  // Will be overwritten by checkpoint
};

let result = compiled.invoke(new_state, config).await?;

// Execution picks up from the last checkpoint
println!("Resumed at step: {}", result.step);

Resume from Specific Checkpoint

You can also resume from a specific checkpoint ID: 

// Get checkpoint history
let checkpoints = checkpointer.list_checkpoints(thread_id).await?;

// Resume from second-to-last checkpoint
if checkpoints.len() > 1 {
    let checkpoint_id = &checkpoints[checkpoints.len() - 2].id;

    let config = CheckpointConfig::new()
        .with_thread_id(thread_id)
        .with_checkpoint_id(checkpoint_id);

    let result = compiled.invoke(initial_state, config).await?;
}

Time Travel Debugging

Checkpoints enable time-travel debugging - you can step backwards through execution: 

// List all checkpoints for a thread
let checkpoints = checkpointer.list_checkpoints("thread-123").await?;

println!("Execution history:");
for (i, checkpoint) in checkpoints.iter().enumerate() {
    println!("  {}. Step {}: {} messages",
        i,
        checkpoint.metadata.step,
        checkpoint.state.messages.len()
    );
}

// Replay from any checkpoint
println!("\nReplaying from step 3:");
let config = CheckpointConfig::new()
    .with_thread_id("thread-123")
    .with_checkpoint_id(&checkpoints[3].id);

let replayed_state = compiled.invoke(initial_state, config).await?;

Branching Workflows

Create alternative execution paths from any checkpoint: 

// Original execution
let thread_a = "thread-main";
let config_a = CheckpointConfig::new().with_thread_id(thread_a);
let result_a = compiled.invoke(initial_state.clone(), config_a).await?;

// Get checkpoint from step 3
let checkpoints = checkpointer.list_checkpoints(thread_a).await?;
let step_3_checkpoint = &checkpoints[3];

// Create a branch from step 3
let thread_b = "thread-branch";
let config_b = CheckpointConfig::new()
    .with_thread_id(thread_b)
    .with_checkpoint_id(&step_3_checkpoint.id);

// Different input for the branch
let branch_state = State {
    messages: vec![Message::human("Take a different path")],
    step: 0,  // Loaded from checkpoint
};

let result_b = compiled.invoke(branch_state, config_b).await?;

// Now you have two timelines:
// thread-main: Original execution
// thread-branch: Alternative path from step 3

Use Case: This is perfect for A/B testing different prompts, trying alternative tool configurations, or exploring different reasoning paths.

Multi-User Support

Use unique thread IDs to support multiple concurrent users: 

// User A's conversation
let user_a_thread = "user-alice-session-1";
let config_a = CheckpointConfig::new().with_thread_id(user_a_thread);

let state_a = State {
    messages: vec![Message::human("Hello from Alice")],
    step: 0,
};
let result_a = compiled.invoke(state_a, config_a).await?;

// User B's conversation (completely independent)
let user_b_thread = "user-bob-session-1";
let config_b = CheckpointConfig::new().with_thread_id(user_b_thread);

let state_b = State {
    messages: vec![Message::human("Hello from Bob")],
    step: 0,
};
let result_b = compiled.invoke(state_b, config_b).await?;

// Each user has their own checkpoint history
// Conversations never interfere with each other

Thread Naming Conventions

Pattern Use Case Example
user-{id}-session-{n} Multi-user sessions user-123-session-1
workflow-{type}-{id} Different workflow types workflow-approval-789
{date}-{task} Time-based organization 2025-01-15-report

Best Practices

1. Clean Up Old Checkpoints

Checkpoints accumulate over time. Implement cleanup for old threads: 

// Delete checkpoints older than 30 days
checkpointer.delete_old_checkpoints(
    Duration::from_days(30)
).await?;

2. Use Descriptive Thread IDs

Make thread IDs self-documenting so you can identify conversations later: user-123-support-ticket-456 instead of just abc123

3. Handle Checkpoint Not Found

Always handle the case where a checkpoint doesn't exist: 

match compiled.invoke(state, config).await {
    Ok(result) => Ok(result),
    Err(GraphError::CheckpointNotFound(_)) => {
        // Start fresh if checkpoint missing
        compiled.invoke(initial_state, config).await
    }
    Err(e) => Err(e),
}

4. Monitor Checkpoint Size

Large states slow down checkpointing. Keep state minimal by:

  • Storing only essential data in state
  • Trimming old messages from history
  • Moving large blobs to external storage (S3, etc.)
